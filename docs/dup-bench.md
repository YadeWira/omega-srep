# Omega SREP `-dup` benchmark (F5.4)

Generated by `tests/dup_bench.sh` on 2026-04-27.

## Setup

- Host: `Linux 6.12.74+deb13+1-amd64 x86_64`, 4 cores, 7.0 GiB RAM
- Method: `-m4`
- CDC buffer size: 8 MiB (`--chunk-buf=8388608`)
- Block content: 1 MiB text + 31 MiB xorshift64 binary, repeated identically

RSS measurements use `tests/measure.py` (a Python wrapper around
`getrusage(RUSAGE_CHILDREN)`, since GNU `time` isn't installed on
this host). **Config B uses the native `-dup` flag** (F5.3c streaming
end-to-end), so the numbers below include the dedup pre-pass and
post-pass — peak RAM is independent of input size.

| corpus  | A compress RSS | B compress RSS | A decompress RSS | B decompress RSS | A wall (compress) | B wall (compress) |
|--------:|---------------:|---------------:|-----------------:|-----------------:|------------------:|------------------:|
| 128 MiB | 21820 KiB      | 19208 KiB      | 53636 KiB        | 18436 KiB        | 0.44 s            | 0.83 s            |
| 256 MiB | 25916 KiB      | 19288 KiB      | 53636 KiB        | 18948 KiB        | 0.93 s            | 1.44 s            |
| 512 MiB | 34108 KiB      | 19444 KiB      | 53636 KiB        | 19972 KiB        | 1.75 s            | 2.68 s            |

B's RSS is essentially flat across input size: the streaming dedup
encoder reads input in 8 MiB buffers, writes the body to disk on the
fly, and only retains the chunk table in memory. The chunk table
grows linearly with chunk count (≈ 23 KiB per MiB of input here),
which accounts for the small upward slope (~250 KiB per doubling).
A's compress RSS grows with the input because SREP holds the
match-tracking state proportional to the dictionary view; A's
decompress RSS is bounded by the SREP buffer cap (`-b8mb`).

Decompress wall-clock is **faster** for B than A from 256 MiB
upward — SREP only ever processes the 32 MiB unique-chunk body, and
ref expansion via `fseek` on an already-written output is cheaper
than running SREP over a longer stream.

## F5.3b → F5.3c history

For the historical record, the original F5.3b implementation loaded
the input into a contiguous `std::vector` for `encode_split`, so
end-to-end peak RSS on the same 128 MiB corpus was 198196 KiB
compress / 214924 KiB decompress. F5.3c rewrote the dedup pre/post-
pass to stream (per-buffer CDC, seek-based ref expansion on
decode) and dropped peak RAM by ~10× without changing the on-disk
archive — F5.3a, F5.3b, and F5.3c output is byte-identical and the
three paths interoperate.

The streaming encoder trusts the 64-bit chunk hash for dedup
decisions rather than retaining all unique-chunk bytes for
byte-compare on collision (which would defeat the streaming goal);
the design doc accepts the 1e-7-per-million-chunks collision
probability as "acceptable in practice" for content-defined dedup.

## Success criterion

Design target (per `docs/dup-mode-design.md`): B's peak decompress
RSS < A's by ≥50% with archive bloat ≤10%.

| corpus  | decompress RSS reduction | archive size  | both met? |
|--------:|-------------------------:|--------------:|----------:|
| 128 MiB | -65.6%                   | 100.3% of A   | ✓         |
| 256 MiB | -64.7%                   | 100.5% of A   | ✓         |
| 512 MiB | -62.8%                   | 101.0% of A   | ✓         |

The F5 thesis is validated end-to-end on the production native
`-dup` path, across a 4× range of input sizes.

## Reproduction

```
make bin/osrep bin/dedup_test
bash tests/dup_bench.sh                                                   # 128 MiB
OSREP_BENCH_REPEATS=8  OSREP_BENCH_OUT=docs/dup-bench-256.md bash tests/dup_bench.sh
OSREP_BENCH_REPEATS=16 OSREP_BENCH_OUT=docs/dup-bench-512.md bash tests/dup_bench.sh
```

Tunables (env vars): `OSREP_BENCH_BLOCK_MB`, `OSREP_BENCH_REPEATS`,
`OSREP_BENCH_METHOD`, `OSREP_BENCH_BUF`, `OSREP_BENCH_DIR`,
`OSREP_BENCH_OUT`.
